TTL Support

Keys with a TTL in Valkey have that TTL propagated through the entire pipeline: captured at write time, stored in the backend, filtered on read-through, and restored when populating the cache.

Capturing TTL at write time

When the keyspace event handler fires on a SET, it calls PTTL on the key to get the remaining TTL in milliseconds. If positive, it computes an absolute Unix epoch expiry (seconds) by adding the remaining time to the current time. This absolute form is what gets stored in the backend.

Why PTTL instead of EXPIRETIME? PTTL has been available since Valkey's earliest releases, while EXPIRETIME requires 7.0+. We round up to seconds since both sinks store seconds.

Storage format

Each backend stores the TTL differently, both with zero overhead:

BackendStorageFormat
DynamoDBttl attribute{"N": "1700000000"} (epoch seconds)
S3 Expressx-amz-meta-expires-at headerObject metadata header (epoch seconds)

DynamoDB: the ttl attribute is a Number. When present, it can also be configured as DynamoDB's native TTL attribute for automatic background cleanup by the service itself. When absent (no-TTL keys), the attribute is simply omitted.

S3 Express: the TTL is stored as a custom metadata header on the object. S3 metadata headers are included in PUT/GET requests at no extra cost or latency. No body format change means full backward compatibility with existing objects (they just have no TTL metadata, which is treated as "no expiry").

Filtering on read-through

SINK.GET never serves expired data. Three layers of TTL filtering ensure this, even when the backend hasn't cleaned up yet:

  1. Backend-side (sink lookup()): Both the S3 and DynamoDB sink implementations check now >= expires_at immediately after reading the response. If the item is expired, lookup() returns None. The expired data is never sent back to the caller. This catches the common case where the backend still has the item (DynamoDB's native TTL cleanup can lag up to 48 hours; S3 has no automatic expiry at all).
  2. Bridging thread (readthrough.rs): Even if the sink returns a value, the SINK.GET bridging thread rechecks expiry before doing anything with it. Time may have elapsed between the backend read and the thread receiving the result. If now >= expires_at, the thread returns Null to the client and does not populate the cache: 
    if let Some(exp) = expires_at {
    if now_secs >= exp {
        // Data is expired — don't populate cache, return nil.
        thread_ctx.reply(Ok(ValkeyValue::Null));
        return;
    }
}
  3. Cache populate guard: After the expiry check passes, the bridging thread computes remaining = expires_at - now. If this drops to zero during processing (race between check and SET), the thread skips the SET and returns Null. This prevents caching a value with zero or negative TTL.

The net effect: SINK.GET is a strict TTL filter. Even if the backend stores stale data indefinitely, the client never sees it.

Cache populate with TTL

When populating the local cache from a read-through result, the remaining TTL is computed as expires_at - now. If positive, the cache is populated with SET key value EX remaining. If zero or negative (expired during processing), the value is not cached and nil is returned.

DynamoDB native TTL. If you configure the ttl attribute as DynamoDB's TTL attribute (aws dynamodb update-time-to-live --table-name ... --time-to-live-specification Enabled=true,AttributeName=ttl), DynamoDB will automatically delete expired items in the background. This keeps the table clean without any application-side logic. The module's read-side filtering ensures expired items are never served even before DynamoDB's background cleanup runs (which can have up to 48 hours of delay).

Replication

Valkey supports primary-replica replication. On replicas, write-behind should be disabled (the primary handles persistence), but read-through should still work (replicas can serve cached data from the backend).

Role detection

The module tracks the current role in a static AtomicBool:

static IS_PRIMARY: AtomicBool = AtomicBool::new(false);

// Fires on failover/promotion via distributed_slice.
#[distributed_slice(ROLE_CHANGED_SERVER_EVENTS_LIST)]
fn on_role_changed(_ctx: &Context, new_role: ServerRole) {
    IS_PRIMARY.store(new_role == ServerRole::Primary, Ordering::Relaxed);
}

Subscribe-then-check ordering

The valkey_module! macro calls register_server_events() before init(). This means the role-change event handler is subscribed before the module first checks the current role in init(). There is no window where a role change could be missed. The initial role is read from ctx.get_flags().contains(ContextFlags::MASTER) in init().

What happens on replicas

  • Write-behind disabled: The keyspace event handler returns immediately when !IS_PRIMARY. No writes are enqueued to the channel, no backend writes occur.
  • SINK.GET works normally: Read-through still queries the backends and returns the value to the client.
  • Cache populate fails silently: The SET to populate the local cache returns READONLY on replicas. The let _ = guard.call("SET", ...) silently ignores this. The client still gets the value from the backend.

Data loss window

If the primary crashes, up to one debounce window (default 1 second) of writes may be lost. This is the same tradeoff as any write-behind system. The replica, if promoted, begins accepting writes and persisting them from that point forward.

SINK.GET on replicas does not cache. Because the SET to populate the cache fails with READONLY, every SINK.GET on a replica hits the backend. This is by design: replicas should not have independent caches that diverge from the primary. If SINK.GET read-through latency on replicas is a concern, consider using READONLY mode in your client library to route reads to the primary.

Corner Cases and Gotchas

@STRING vs @SET

The valkey_module! macro's event handler registration uses @ prefixes for event types. @SET does not mean the SET command. It means the Set data type (SADD, SREM, SPOP). The correct subscription for string SET commands is @STRING.

// WRONG: subscribes to Set data type events (SADD, etc.)
event_handlers: [
    [@SET: keyspace_event_handler]
],

// CORRECT: subscribes to string commands (SET, SETEX, etc.)
event_handlers: [
    [@STRING: keyspace_event_handler]
],

This was discovered during live testing when writes_coalesced stayed at 0 despite keyspace notifications being enabled.

Auto-enabling keyspace notifications

Valkey defaults notify-keyspace-events to an empty string. Without the K$ flags, the module's event handler will never fire. The module auto-enables these during init, reading the current config and preserving existing flags:

crates/sink/src/lib.rs:131-154 
if !current.contains('K') || !current.contains('$') {
    let mut flags = current.clone();
    if !flags.contains('K') { flags.push('K'); }
    if !flags.contains('$') { flags.push('$'); }
    let _ = ctx.call("CONFIG", &["SET", "notify-keyspace-events", &flags]);
}

What's Left

No project is ever truly done. Here are the gaps we know about. If any of them sound interesting to you, they'd each make a great first contribution.

DEL propagation

DEL commands are not captured. When a key is deleted from Valkey, it remains in the backend (DynamoDB or S3) indefinitely. The fix would be to subscribe to del and expired events in the keyspace handler and send a new message type (e.g. KeyDeleted) to the background runtime, which would call DeleteObject or DeleteItem on the sinks. For S3, consider using S3 lifecycle rules as a simpler alternative for eventual cleanup.

Binary value fidelity in read-through

The read-through path in readthrough.rs replies with ValkeyValue::StringBuffer(value) (binary-safe), but earlier versions used String::from_utf8_lossy. While the current reply path is binary-safe, any logging or debugging code that converts values to strings should be audited to ensure non-UTF8 binary values are not silently corrupted.

Integration tests

All 96 tests are unit tests. There are no integration tests against a running Valkey server, real AWS endpoints, or even LocalStack. The unit tests are thorough (they cover coalescing edge cases, config parsing, base64 roundtrips, XML parsing, circuit breaker state machines, retry logic) but they cannot test the end-to-end flow: module load, SET, keyspace event, coalesce, flush, SINK.GET, read-through, cache populate.

Sanity test refactoring

tools/loadtest/src/main.rs is 1,305 lines in a single file. It handles Docker management, configuration parsing, burst testing, sustained testing, metrics collection, and output formatting. This should be split into modules (e.g. config.rs, docker.rs, burst.rs, sustained.rs, util.rs) for maintainability. (Despite the directory name, these are low-load sanity tests, not production load tests.)

Release profile optimization

The workspace Cargo.toml has no [profile.release] section. Adding LTO (lto = true), codegen-units = 1, and strip = true would produce a smaller, faster binary at the cost of longer compile times. For a module loaded once at server startup, this tradeoff is almost always worth it.

ConfigDelta wiring

The ConfigUpdate(ConfigDelta) message type exists in channel.rs, the background loop handles it in background.rs (with apply_config_delta supporting debounce, size threshold, and read-through toggling), but no code path ever sends a ConfigUpdate message. The #[allow(dead_code)] on the enum confirms this. To wire it up, the module would need to register a CONFIG SET handler (or a custom command like SINK.CONFIG) that creates and sends the appropriate ConfigDelta variants.

Credential refresh error handling

The RefreshableCredentials provider in aws.rs refreshes from environment variables. If the environment variables are removed or changed while the module is running (e.g. rotated IAM credentials on an EC2 instance), the refresh will fail with a hard error. Consider falling back to IMDS (Instance Metadata Service) for EC2 environments, or at minimum logging a warning when env-based refresh fails after previously succeeding.

Zero-copy keyspace access

The keyspace event handler currently calls GET to fetch the value after every SET, then clones the bytes into a Vec<u8> for the channel message. The same clone happens on the read-through path when populating the cache. The Valkey Module API exposes ValkeyModule_OpenKey + ValkeyModule_StringDMA, which returns a direct pointer to the key's internal string buffer without copying. Using DMA access in the event handler would eliminate one allocation and one memcpy per write. On the read-through side, ValkeyModule_StringSet with a pre-allocated buffer could avoid the intermediate Vec. For high-throughput workloads with large values, this could meaningfully reduce memory pressure and improve cache-line efficiency on the main thread.

Conclusion

Over the course of this series, we built a transparent write-behind and read-through persistence layer for Valkey in roughly 2,700 lines of Rust. No SDK bloat, no external sidecars, no changes to application code beyond swapping GET for SINK.GET.

The key wins:

  • Zero main-thread impact. Every network call happens on background threads. The keyspace event handler does a lock-free try_send and returns in nanoseconds. Valkey's single-threaded event loop never stalls.
  • Size-based routing. Small values go to DynamoDB for its per-WCU pricing advantage. Large values go to S3 Express for its flat per-request cost. The router makes this invisible to the application.
  • TTL fidelity. Expiry times propagate through the entire pipeline: captured at write time, stored in the backend, filtered on read-through, and restored when populating the cache. Expired data is never served, even when the backend hasn't cleaned up yet.
  • Replica awareness. Write-behind runs on primaries only. Replicas do read-through but don't duplicate writes. Role changes are detected via distributed_slice event handlers registered before init(), so there is no window where a failover could be missed.

But the module itself isn't really the point. The module system is the takeaway. Valkey is extensible in ways that most people don't realize. A shared library loaded at startup gets the same in-process performance as built-in commands, full access to the keyspace, and the ability to offload arbitrary async work to background threads. The patterns in valkey-sink (crossbeam channels for non-blocking dispatch, BlockClient for async read-through, distributed_slice for event subscription, ThreadSafeContext for bridging async to sync) apply to any module you might build: CDC pipelines, custom data structures, real-time analytics, domain-specific caching, audit logging.

If you want to get started, the path is short:

  1. Clone the valkeymodule-rs repo and build hello.rs. Two minutes to a working module.
  2. Subscribe to a keyspace event. Now you can react to writes as they happen.
  3. Add a background thread for async work. Now you can do anything without blocking the event loop.

That's the whole surface area. Fork the repo, break something, fix it, and build something that doesn't exist yet. The Valkey module ecosystem is still young, and there is a lot of room to build.
