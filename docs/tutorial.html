What Are Valkey Modules?

Valkey is an open-source in-memory data store maintained by the Linux Foundation. One of its most powerful features is loadable modules: shared libraries (.so on Linux, .dylib on macOS) that extend the server with custom commands, custom data types, and event handlers. If you've ever wished Valkey had a feature it doesn't, modules are how you add it.

Modules run in-process with the server. They share the same address space, can call native Valkey commands (GET, SET, PTTL, CONFIG) as local function calls, and can subscribe to keyspace notifications. This is fundamentally different from an external sidecar process that connects over TCP: there is zero network hop for internal operations, and command execution completes in nanoseconds rather than microseconds. That means your custom logic runs at the same speed as built-in Valkey commands.

Why modules instead of external processes?

Before we get into the details, it's worth understanding why we chose the module approach at all. There are three ways to extend Valkey with custom logic, and the tradeoffs are significant:

ApproachRead latencyWrite overheadDeployment
External sidecarNetwork RTT (~100us+)Two hops per writeSeparate process
Lua scriptIn-process, but interpretedNo async I/OLoaded per-connection
Module (.so)In-process, native speedZero-copy keyspace accessLoaded once at startup

For a write-behind persistence system, the module approach is the clear winner. Keyspace events are delivered synchronously in the same thread as the SET command. The value can be read via a local GET call in nanoseconds. And the only network I/O (to S3 or DynamoDB) happens asynchronously on background threads. An external sidecar would need to subscribe to keyspace notifications over a TCP connection, adding latency and a failure mode on every single write. A Lua script can't do async I/O at all.

Notable Valkey modules for context. valkey-bloom (bloom filters), valkey-json (JSON document storage), valkey-search (full-text and vector search). All follow the same pattern: a C-compatible shared library loaded via MODULE LOAD or --loadmodule. valkey-sink is smaller and more focused: two commands, one event subscription, ~2,700 lines. The point is: you don't need to build something massive. A focused module that does one thing well is the sweet spot.

Key concepts

If you haven't worked with Valkey modules before, here are the four building blocks you'll see throughout this article:

  • Context (&Context): the handle to the running Valkey server. Every command handler and event handler receives a Context, through which it can call native commands, read server flags, block clients, and log messages.
  • Commands: functions registered at module load time. They have a name (e.g. SINK.GET), a handler function, and metadata about which arguments are keys (for cluster routing). When a client sends the command, Valkey dispatches to the handler.
  • Keyspace notifications: event subscriptions. A module can register a handler that fires on every string write (@STRING), every set operation (@SET), every expired key (@EXPIRED), etc. The handler receives the event type, the event name (e.g. "set"), and the key bytes.
  • BlockClient: an API that lets a command handler suspend the client without blocking the event loop. The client is put on hold, the event loop continues serving other clients, and a background thread or task eventually calls ThreadSafeContext::reply() to unblock the client with a response.

Writing Valkey Modules in Rust

The valkeymodule-rs crate (published as valkey-module on crates.io) provides safe Rust bindings for the Valkey Module API. It handles the unsafe C FFI, memory management, and type conversions so you can write idiomatic Rust without touching raw pointers. The barrier to entry is genuinely low. If you can write a Rust function, you can write a Valkey command.

The valkey_module! macro

Every Valkey module written in Rust uses this macro as its entry point. It generates the C-compatible ValkeyModule_OnLoad function that Valkey expects, registers commands, event handlers, and the custom allocator. Here is a minimal "hello world" module:

use valkey_module::{
    alloc::ValkeyAlloc, valkey_module,
    Context, ValkeyResult, ValkeyString, ValkeyValue, Status,
};

fn init(_ctx: &Context, _args: &[ValkeyString]) -> Status {
    Status::Ok
}

fn hello_command(_ctx: &Context, _args: Vec<ValkeyString>) -> ValkeyResult {
    Ok(ValkeyValue::SimpleStringStatic("hello from Rust!"))
}

valkey_module! {
    name: "hello",
    version: 1,
    allocator: (ValkeyAlloc, ValkeyAlloc),
    data_types: [],
    init: init,
    commands: [
        ["HELLO.SAY", hello_command, "readonly", 0, 0, 0],
    ],
    event_handlers: [],
}

Build it with cargo build --release, load it with valkey-server --loadmodule ./target/release/libhello.so, and call HELLO.SAY from any client.

How commands work

Every command handler has the signature fn(ctx: &Context, args: Vec<ValkeyString>) -> ValkeyResult. The first element of args is always the command name itself (e.g. SINK.GET). The return type ValkeyResult is Result<ValkeyValue, ValkeyError>. ValkeyValue has variants for strings, integers, arrays, nil, and NoReply (used with BlockClient).

How event handlers work

Event handlers are registered in the event_handlers array of the macro. The @ prefix denotes the event type, not the command name:

SubscriptionFires onCommon confusion
@STRINGSET, SETEX, SETNX, MSET, APPEND, ...This is what you want for SET commands
@SETSADD, SREM, SPOP, ...NOT the SET command: this is the Set data type
@HASHHSET, HDEL, ...Hash data type operations
@EXPIREDKey expiration eventsValue is already gone by the time the handler fires

The allocator constraint

The allocator: (ValkeyAlloc, ValkeyAlloc) line tells Rust to use Valkey's built-in allocator (jemalloc, by default) for all heap allocations. This is essential: Valkey tracks memory usage across all modules, and using the system allocator would produce inaccurate INFO memory output. However, ValkeyAlloc requires a running Valkey server to function. For unit tests, the crate provides an enable-system-alloc feature flag that swaps in the standard system allocator:

# Unit tests use system allocator since ValkeyAlloc requires a running server
cargo test --features enable-system-alloc

The crate type: cdylib

The Cargo.toml must declare crate-type = ["cdylib"]. This tells the Rust compiler to produce a C-compatible dynamic library (.so / .dylib) instead of a Rust-native .rlib. Valkey's module loader expects a C ABI entry point.

The linkme crate and distributed_slice

Server event handlers (like role-change notifications) are registered using the linkme crate's #[distributed_slice] attribute. This is a compile-time mechanism: the handler function is placed into a linker section, and the valkey_module! macro iterates over all entries in that section during module load. The key property is that event handlers are registered before init() runs, so there is no window where a role change could be missed.

The main thread is single-threaded and synchronous. You CANNOT .await anything on the Valkey main thread. Keyspace event handlers, command handlers, and init all run on this thread. Any async work (HTTP calls, file I/O, timers) must be offloaded to a background runtime. This constraint shapes the entire architecture of valkey-sink, and it's a good thing. Once you internalize it, the design of any Valkey module becomes straightforward: fast synchronous logic on the main thread, everything else in the background. The crossbeam channel, the bridging thread for BlockClient, and the separate tokio runtime all follow directly from this one rule.

Go Build Your Own

If you've read this far, you know enough to build a Valkey module. Seriously. The entire surface area is:

  1. A valkey_module! macro call that registers your commands and event handlers.
  2. Command handlers: fn(ctx: &Context, args: Vec<ValkeyString>) -> ValkeyResult.
  3. Event handlers: fn(ctx: &Context, event_type: NotifyEvent, event: &str, key: &[u8]).
  4. One rule: don't block the main thread.

That's it. Here's a concrete path to your first module:

  1. Clone the hello.rs example. Build it, load it into a running Valkey server, call your command from valkey-cli. The whole loop takes about two minutes.
  2. Add a second command. Maybe one that reads a key and transforms the value. Now you have a module that does something useful.
  3. Subscribe to an event. The events.rs example shows how. Once you can react to writes as they happen, the design space opens up: audit logging, real-time streaming, change data capture, custom replication. All become straightforward.
  4. Need async work? Look at block.rs and threads.rs for the BlockClient + ThreadSafeContext pattern. valkey-sink uses exactly this pattern for SINK.GET read-through.

The valkeymodule-rs repository has 36 examples covering commands, data types, events, timers, ACLs, streams, configuration, and more. Between those examples and this walkthrough of a real-world module, you have everything you need.

The Valkey module ecosystem is still young. There's a lot of low-hanging fruit: custom data structures, integrations with message queues, CDC pipelines, domain-specific caching strategies, observability hooks. If you build something interesting, the community would love to hear about it at github.com/valkey-io.
