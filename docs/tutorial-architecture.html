Extending Valkey's Working Set

Valkey is fast because everything lives in memory. But memory is finite and expensive. When your working set grows beyond what a single instance can hold, you have a few options: shard across multiple instances, evict cold keys, or extend Valkey with a durable backing store that transparently handles overflow. valkey-sink takes the third approach.

The idea: every write to Valkey is asynchronously persisted to an external store. On a cache miss, SINK.GET transparently reads from the backend and repopulates the cache. From the application's perspective, Valkey's effective working set becomes the union of in-memory data and whatever is in the backing store, without changing a single line of application code beyond swapping GET for SINK.GET.

For backends, valkey-sink currently has two sinks: DynamoDB for smaller values (below a configurable threshold, default 4KB) and S3 Express One Zone for larger values. DynamoDB gives single-digit millisecond latency with per-WCU pricing that favors small items; S3 Express gives single-digit millisecond latency at any object size with flat per-request pricing. A SinkRouter partitions writes by value size and dispatches to the right backend. The Sink trait (write_batch + lookup) is pluggable. Adding a new backend means implementing two methods.

Requirements

  1. Transparent write-behind. Every SET is asynchronously persisted. No application code changes on the write path. Hot keys updated many times per second should be coalesced: only the final value in a debounce window gets flushed.
  2. Size-based routing. Small values go to DynamoDB, large values go to S3 Express. The threshold is configurable. Both sinks flush concurrently within a batch.
  3. Read-through on cache miss. SINK.GET checks the backend when Valkey returns nil, repopulates the cache, and returns the value, all invisible to the caller.
  4. TTL fidelity. Keys with an expiry in Valkey keep that expiry through the backend and back. Expired data is never served.
  5. Replica awareness. Write-behind runs on primaries only. Replicas still do read-through but don't duplicate writes.
  6. Zero impact on Valkey's main thread. Valkey is single-threaded. A 5ms network call in a keyspace handler stalls every client, and a cache miss can't .await anything. The module must not add measurable latency to SET or GET; all network I/O happens on background threads, and read-through uses BlockClient to suspend the caller while the event loop continues. The Non-Blocking Constraint

This is the fundamental constraint that shapes everything else. Valkey is single-threaded: every microsecond spent in a keyspace event handler is a microsecond where no other client can be served. Our module needs to intercept every SET, capture the key and value, and hand it off for persistence, all without adding measurable latency to the SET itself. The solution is a producer-consumer pattern with a bounded crossbeam channel:

Main thread (single-threaded)       Background (multi-threaded tokio)
+----------------------------+      +------------------------------+
| SET key value              |      |                              |
|   |                        |      |  crossbeam::Receiver         |
|   v                        |      |    |                         |
| keyspace_event_handler()   |      |    v                         |
|   |                        |      |  CoalesceMap.upsert(k, v)    |
|   v                        |      |    |                         |
| channel.try_send(msg)  ----+----->|  timer.tick()                |
|   |                        |      |    |                         |
|   v                        |      |    v                         |
| returns immediately        |      |  drain_ready() -> flush()    |
+----------------------------+      +------------------------------+

The channel implementation details (why crossbeam over tokio channels, why try_send over send, the full keyspace event handler, and the message types) are in channel.rs and lib.rs. See READING-ORDER.md for a guided path through every source file.

Write Coalescing

Without coalescing, a hot key updated 100 times per second would produce 100 S3 PutObject calls per second. That's expensive, slow, and pointless. Only the final value matters. The coalesce map solves this with a debounce window: updates within the window overwrite each other, and only the last value is flushed to the backend. It's a HashMap<Vec<u8>, PendingWrite> where each entry holds the latest value and a first_seen timestamp.

crates/sink/src/coalesce.rs:19-26 
pub struct PendingWrite {
    pub value: Vec<u8>,
    /// Absolute expiry time as Unix epoch seconds. None means no expiry.
    pub expires_at: Option<u64>,
    /// When this key first entered the map. Preserved across upserts so the
    /// debounce deadline is anchored to the first write, not the last.
    pub first_seen: Instant,
}

How it works

Time 0ms:    SET counter 1   ->  upsert("counter", "1", ttl=None, first_seen=0)
Time 10ms:   SET counter 2   ->  upsert("counter", "2", ttl=None, first_seen=0)   value overwritten
Time 100ms:  SET counter 3   ->  upsert("counter", "3", ttl=None, first_seen=0)   first_seen preserved
...
Time 1000ms: timer tick       ->  drain_ready() finds "counter"
                              ->  flush("counter", "3") to backend       1 write, not 100
crates/sink/src/coalesce.rs:49-61 
pub fn upsert(&mut self, key: Vec<u8>, value: Vec<u8>, expires_at: Option<u64>) {
    self.pending
        .entry(key)
        .and_modify(|pw| {
            pw.value = value.clone();
            pw.expires_at = expires_at;
        })
        .or_insert(PendingWrite {
            value,
            expires_at,
            first_seen: Instant::now(),
        });
}

The and_modify closure updates the value and TTL. first_seen is untouched on update so the debounce window stays anchored. On insert (or_insert), it records the current time.

Draining ready entries

crates/sink/src/coalesce.rs:65-82 
pub fn drain_ready(&mut self) -> Vec<(Vec<u8>, Vec<u8>, Option<u64>)> {
    let now = Instant::now();
    let window = self.debounce_window;

    let mut ready = Vec::new();
    self.pending.retain(|k, pw| {
        if now.duration_since(pw.first_seen) >= window {
            let value = std::mem::take(&mut pw.value);
            ready.push((k.clone(), value, pw.expires_at));
            false  // remove from map
        } else {
            true   // keep in map
        }
    });
    ready
}

Uses retain to avoid cloning keys: ready entries are extracted in-place. The value is moved out via std::mem::take (swaps with an empty Vec) to avoid a clone.

Re-enqueue on failure

If a flush fails, entries are put back into the map. But only if a newer write has not already arrived for the same key:

crates/sink/src/coalesce.rs:97-105 
pub fn re_enqueue(&mut self, entries: Vec<(Vec<u8>, Vec<u8>, Option<u64>)>) {
    for (key, value, expires_at) in entries {
        self.pending.entry(key).or_insert(PendingWrite {
            value,
            expires_at,
            first_seen: Instant::now(),
        });
    }
}

or_insert rather than insert. If the key is already in the map (a new write arrived during the failed flush), the newer value takes priority.

Size-Based Routing

DynamoDB has a 400KB item size limit and charges per write capacity unit (1KB per WCU). S3 has no practical size limit and charges per request regardless of size. So a 1KB value is cheaper in DynamoDB, and a 1MB value is far cheaper in S3. The SinkRouter uses a configurable size threshold (default 4KB) to route each value to the right backend. The router doesn't care what the backends are. It just partitions and dispatches:

crates/sink/src/sink/mod.rs:83-107 
pub fn partition_entries<'a>(
    &self,
    entries: &'a [SinkEntry],
) -> (Vec<&'a SinkEntry>, Vec<&'a SinkEntry>) {
    let mut dynamo_entries = Vec::new();
    let mut s3_entries = Vec::new();

    for entry in entries {
        match (&self.dynamo, &self.s3) {
            (Some(_), Some(_)) => {
                if entry.value.len() <= self.size_threshold() {
                    dynamo_entries.push(entry);
                } else {
                    s3_entries.push(entry);
                }
            }
            (Some(_), None) => dynamo_entries.push(entry),
            (None, Some(_)) => s3_entries.push(entry),
            (None, None) => {}
        }
    }

    (dynamo_entries, s3_entries)
}

The default threshold is 4KB, a reasonable split point where DynamoDB's per-WCU pricing is still competitive and S3 Express's per-request pricing starts to win. Both limits are configurable.

DynamoDB and S3 flushes within a single batch run concurrently via tokio::join!, so flush latency is max(dynamo, s3) rather than dynamo + s3.

DynamoDB Sink

DynamoDB is ideal for small, high-frequency key-value writes: single-digit millisecond latency, automatic scaling, and native TTL support. Like S3, we bypass the SDK and make direct HTTP POST requests to the DynamoDB JSON API, signed with SigV4. The table schema is deliberately minimal:

AttributeTypeDescription
pkB (Binary)Valkey key bytes, base64-encoded
valB (Binary)Valkey value bytes, base64-encoded
tsN (Number)Unix timestamp of the write
ttlN (Number)Absolute expiry (epoch seconds). Omitted when no expiry. Can be configured as a DynamoDB TTL attribute for automatic cleanup.

DynamoDB's BatchWriteItem is limited to 25 items per request. The sink chunks entries into batches of 25 and executes them concurrently via futures::stream::buffer_unordered (default concurrency: 32). Each chunk handles UnprocessedItems internally with up to 5 inner retries and exponential backoff.

crates/sink/src/sink/dynamo.rs:152-244

            let requests: Vec<Value> = chunk.iter()
                .map(|entry| {
                    let mut item = serde_json::Map::new();
                    item.insert("pk",  json!({ "B": b64(&entry.key) }));
                    item.insert("val", json!({ "B": b64(&entry.value) }));
                    item.insert("ts",  json!({ "N": ts.to_string() }));
                    if let Some(exp) = entry.expires_at {
                        item.insert("ttl", json!({ "N": exp.to_string() }));
                    }
                    json!({ "PutRequest": { "Item": item } })
                })
                .collect();

            async move {
                self.call_dynamo("DynamoDB_20120810.BatchWriteItem", &body).await
            }
        })
        .collect();

    let results: Vec<_> = stream::iter(futs)
        .buffer_unordered(self.write_concurrency)
        .collect().await;
    // ...
}

Both keys and values are base64-encoded into DynamoDB Binary attributes. Throttle responses (HTTP 400 with ProvisionedThroughputExceededException or ThrottlingException) are classified as retriable errors so the outer retry policy kicks in.

S3 Express Sink

S3 Express One Zone provides single-digit millisecond latency for any object size, making it the natural choice for values above the DynamoDB threshold. Each value is stored as a separate S3 object via PutObject, with the Valkey key encoded into the object key path.

Session authentication

S3 Express directory buckets use a two-step auth flow that differs from regular S3. First, a CreateSession call (signed with standard IAM credentials via SigV4) returns temporary session credentials with a 5-minute TTL. Then, PutObject and GetObject requests are signed with those session credentials using the x-amz-s3session-token header (not the standard x-amz-security-token) and the s3express service scope.

Session credentials are cached behind a tokio::sync::RwLock with a double-check pattern: the fast path acquires a read lock and returns the cached session if still valid; on expiry, a Mutex serializes refresh attempts so that only one task calls CreateSession while the rest wait and reuse the fresh session. This prevents a thundering herd when many concurrent uploads all discover the session expired at the same time. A 30-second early-refresh buffer (is_expired checks elapsed + 30s >= ttl) avoids using a session that expires mid-request.

Concurrent uploads

S3 writes are parallelized using futures::stream::buffer_unordered with a configurable concurrency limit (default 16). Session credentials are fetched once before the batch starts, so all uploads in a batch share the same session:

crates/sink/src/sink/s3.rs:367-392 
pub async fn write_batch_refs(&self, entries: &[&SinkEntry]) -> Result<(), SinkError> {
    self.get_session_creds().await?;

    let futs: Vec<_> = entries.iter()
        .map(|entry| {
            let obj_key = self.object_key(&entry.key);
            async move { self.put_object(&obj_key, body, expires_at).await }
        })
        .collect();

    let results: Vec<_> = stream::iter(futs)
        .buffer_unordered(self.write_concurrency)
        .collect().await;
    // ...
}

TTL is stored as an x-amz-meta-expires-at object metadata header at zero extra cost. On read-through, the sink checks this header and returns None for expired objects.

The Background Loop

The background runtime is a tokio multi-thread runtime with a configurable number of workers (default 4). Its main loop does three things on each tick: drain the channel, check for ready entries, flush them.

crates/sink/src/background.rs:71-193 
async fn run_loop(
    rx: Receiver<MainToBackground>,
    router: SinkRouter,
    initial_debounce: Duration,
    max_entries: usize,
    retry_policy: RetryPolicy,
    circuit_breaker: CircuitBreaker,
) {
    let mut coalesce = CoalesceMap::new(initial_debounce, max_entries);

    // Tick at half the debounce window.
    let tick_interval = initial_debounce / 2;
    let mut timer = tokio::time::interval(
        tick_interval.max(Duration::from_millis(50))
    );

    loop {
        // 1. Drain all pending channel messages (non-blocking).
        let mut shutdown_requested = false;
        loop {
            match rx.try_recv() {
                Ok(MainToBackground::KeyWritten { key, value, expires_at }) => {
                    metrics::inc(&metrics::get().writes_coalesced);
                    coalesce.upsert(key, value, expires_at);
                }
                // ... ReadThrough, ConfigUpdate, Shutdown ...
                Err(crossbeam_channel::TryRecvError::Empty) => break,
                Err(crossbeam_channel::TryRecvError::Disconnected) => {
                    shutdown_requested = true;
                    break;
                }
            }
        }

        // 2. Force-flush if over capacity.
        if coalesce.is_over_capacity() {
            let all = coalesce.drain_all();
            // ... flush ...
        }

        // 3. Wait for timer tick, then drain ready entries.
        timer.tick().await;

        if !circuit_breaker.is_open() {
            let ready = coalesce.drain_ready();
            if !ready.is_empty() {
                // ... flush with retry ...
            }
        }
    }
}

Timer tick frequency

The flush timer ticks at debounce_window / 2. With a 1s debounce, the timer fires every 500ms. This means entries are flushed within 1.0x to 1.5x the debounce window. We could tick faster for lower latency, but the overhead is not worth it for write-behind use cases.

Shutdown behavior

On shutdown, the loop drains all remaining entries regardless of the debounce window and flushes them. The deinit function in lib.rs sends a Shutdown message and then polls a completion flag for up to 5 seconds, returning as soon as the final flush finishes rather than sleeping unconditionally.

Read-Through with BlockClient

Write-behind is only half the story. The other half is what happens on a cache miss. When a key has been evicted from memory but still exists in S3 or DynamoDB, we want SINK.GET to transparently fetch it, without the caller knowing or caring that the data came from a backend. But a backend lookup takes milliseconds, and we can't block the event loop waiting for it. Valkey's BlockClient API solves this: it lets a command handler say "I'm not done yet, I'll reply later" while the event loop continues serving other clients. Here's the flow:

 1. Client sends SINK.GET key
 2. Command handler calls native GET  ->  nil (cache miss)
 3. Handler calls ctx.block_client()  ->  client is suspended
 4. Handler sends ReadThrough message through channel
 5. Handler returns ValkeyValue::NoReply (event loop continues)

 6. Background runtime receives ReadThrough message
 7. Looks up key in DynamoDB, then S3 if not found
 8. Sends (value, expires_at) back via oneshot channel

 9.  Bridging thread receives result
10.  Checks TTL: if expired, returns nil
11.  Sets SKIP_ENQUEUE flag (prevents re-upload to backend)
12.  Populates cache with SET + EX (future GETs are fast)
13.  Replies to blocked client via ThreadSafeContext
crates/sink/src/readthrough.rs:17-154 
pub fn sink_get_command(ctx: &Context, args: Vec<ValkeyString>) -> ValkeyResult {
    // Steps 1-4: GET, check cache, block client, send ReadThrough msg
    // ...

    // Step 5: Bridging thread -- converts async oneshot to sync reply.
    std::thread::spawn(move || {
        let thread_ctx = ThreadSafeContext::with_blocked_client(blocked_client);

        match rx.blocking_recv() {
            Ok(Some((value, expires_at))) => {
                metrics::inc(&metrics::get().read_through_hits);

                // Check if the data is already expired.
                let now_secs = /* SystemTime epoch seconds */;
                if let Some(exp) = expires_at {
                    if now_secs >= exp {
                        thread_ctx.reply(Ok(ValkeyValue::Null));
                        return;
                    }
                }

                // Populate cache with binary-safe SET using &[&[u8]] args.
                // Set SKIP_ENQUEUE so the keyspace handler doesn't re-upload.
                {
                    let guard = thread_ctx.lock();
                    crate::SKIP_ENQUEUE.with(|f| f.set(true));

                    let set_result = if let Some(exp) = expires_at {
                        let remaining = exp.saturating_sub(now_secs);
                        if remaining > 0 {
                            guard.call("SET", &[key, value, b"EX", remaining_str])
                        } else {
                            crate::SKIP_ENQUEUE.with(|f| f.set(false));
                            thread_ctx.reply(Ok(ValkeyValue::Null));
                            return;
                        }
                    } else {
                        guard.call("SET", &[key, value])
                    };

                    if set_result.is_err() {
                        crate::SKIP_ENQUEUE.with(|f| f.set(false));
                    }
                }

                // Reply with raw bytes (binary-safe).
                thread_ctx.reply(Ok(ValkeyValue::StringBuffer(value)));
            }
            Ok(None) | Err(_) => {
                metrics::inc(&metrics::get().read_through_misses);
                thread_ctx.reply(Ok(ValkeyValue::Null));
            }
        }
    });

    Ok(ValkeyValue::NoReply)
}

Why a bridging thread?

The background runtime is async (tokio). The ThreadSafeContext::reply() API is synchronous. We cannot call blocking_recv() on a tokio task without risking deadlock (it would block a tokio worker). The bridging thread does zero I/O. It exists only to convert the async oneshot into a synchronous reply call. It lives for the duration of one lookup, typically 5-50ms.

Tradeoffs

Every design decision involves giving something up. Here's an honest accounting of what we traded away and why we think the tradeoffs are worth it.

Eventually consistent, not strongly consistent

SET + sink write is not atomic. There is a window (up to the debounce interval) where data exists only in Valkey's memory. A crash during this window loses the last debounce window of writes. We chose this deliberately: strong consistency would require synchronous writes on the main thread, which violates the non-blocking constraint. For a write-behind cache, eventual consistency is the right tradeoff.

DEL is not propagated

DEL commands are not currently captured. A deleted key remains in the backend until overwritten. This could be added by subscribing to the del event in the keyspace handler.

TTL and expiration

TTL is propagated to the backend when a key is SET with an expiry. However, keys that expire via Valkey's built-in eviction (the @EXPIRED event) are not handled. By the time the expiration event fires, the value is already gone from Valkey's memory and cannot be read. The backend retains the value until its own TTL-based cleanup runs (DynamoDB native TTL, or manual S3 lifecycle rules).

Replica cache populate

On replicas, SINK.GET reads from the backend but cannot populate the local cache (the SET fails with READONLY). Every SINK.GET on a replica hits the backend. This is intentional: replicas should not have independent caches that diverge from the primary.

Binary values in read-through

The read-through path uses String::from_utf8_lossy when calling SET to repopulate the cache. Non-UTF8 binary values will have invalid bytes replaced with the Unicode replacement character. For string values, this is correct. For raw binary, it is lossy.
